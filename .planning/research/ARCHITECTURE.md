### Component Responsibilities

| Component | Responsibility | Typical Implementation |
|-----------|----------------|------------------------|
| API Security | Validates JWTs, handles CORS, and manages endpoint permissions. | Spring Security Filter Chain |
| Domain Modules| Orchestrates business logic for specific features (Catalog, Order). | Service-level Beans with @Service |
| Entities | Defines the data structure and mapping to database tables. | JPA Entities (@Entity, @Table) |
| Repositories | Handles all database interactions and data fetching. | Spring Data JPA @Repository |
| Controllers | Exposes REST endpoints and maps request payloads to DTOs. | @RestController with MapStruct |

```
??? db/migration/     # Liquibase or Flyway SQL scripts

```

### Structure Rationale

- **modules/:** Package-by-feature allows for high cohesion and loose coupling; it also makes future extraction into microservices easier.
- **common/:** Prevents code duplication for cross-cutting concerns like pagination, error responses, and base domain objects.

## Architectural Patterns

### Pattern 1: Modular Monolith

**What:** Encapsulating domain logic into distinct, independent modules within a single deployment unit.
**When to use:** Start-ups, MVPs, or any scenario where the overhead of microservices exceeds the team's capacity.
**Trade-offs:** Faster development and easier deployment, but harder to scale individual components independently.

### Pattern 2: DTO (Data Transfer Object)

**What:** Separating the internal domain entity from the public API contract using DTOs and Mappers (MapStruct).
**When to use:** Always. Avoid exposing JPA entities directly to the client.
**Trade-offs:** Adds some boilerplate but prevents security leaks and unintended data mutations.

### Pattern 3: Optimistic Concurrency Control

**What:** Using @Version on JPA entities to prevent race conditions during concurrent updates (e.g., inventory deduction).
**When to use:** Inventory updates, balance deductions, order status changes.
**Trade-offs:** Retries may be needed if collisions occur, but it's more scalable than pessimistic locking.

## Scaling Considerations

| Scale | Architecture Adjustments |
|-------|--------------------------|
| 0-10k users | Monolith with single DB instance is sufficient. |
| 10k-100k users | Introduce Redis for session caching and read replicas for PostgreSQL. |
| 100k+ users | Consider splitting high-load modules (e.g., Search, Orders) into separate microservices. |

### Scaling Priorities

1. **First bottleneck:** Database read performance (fix with indexes, caching, or read replicas).
2. **Second bottleneck:** Heavy background tasks (fix with asynchronous processing using @Async or message queues like RabbitMQ).

## Anti-Patterns

### Anti-Pattern 1: The "God" Service

**What people do:** Putting all logic into one massive service class.
**Why it's wrong:** Impossible to test or maintain; violates SRP (Single Responsibility Principle).
**Do this instead:** Break down logic into small, focused service beans.

### Anti-Pattern 2: Eager Fetching Everything

**What people do:** Setting all JPA relationships to FetchType.EAGER.
**Why it's wrong:** Causes the N+1 problem and massive memory consumption.
**Do this instead:** Use FetchType.LAZY and explicit JOIN FETCH or Entity Graphs when needed.
